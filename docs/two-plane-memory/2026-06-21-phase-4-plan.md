# Phase 4 — Retrieval & Cross-Chat — Plan (for approval)

**Status:** PROPOSED — no build yet.

**Goal:** Memories that are *written* (Phases 1–3) become *retrieved* — by-meaning, witness-correct, across chats in the same world.

**Decisions locked (user):** ST **Vector Storage is ON** (vector top-k is live once the book is active); STMB **auto-attaches the `<World> - Memories` book per-world**.

**Nature of this phase (important):** unlike Phases 1–3 (pure host-readable logic, unit-TDD), Phase 4 depends on **ST core WI/Vector internals that are NOT readable from the host** (in the container). So it is a **live-integration phase**: discover the API from readable third-party extensions, wire it, and **verify in the running ST** (not just unit tests). Expect more live back-and-forth than 1–3.

**Key reframe from the understanding pass:** the retrieval *pipeline* (vector top-k → witness-filter → budget) is **already delegated to ST** (entries are `vectorized:true` + `characterFilter`-gated). The real gaps are: (a) the world book is **never made active** in ST's WI set, so memories are written but **not injected/recalled** even same-chat; (b) no cross-chat auto-load; (c) no pre-switch flush; (d) tier-bias.

## 4a — World-book activation + retrieval verification (FOUNDATIONAL)

Without this, nothing retrieves. Covers X1 (recall across chats), X3 (world-bound auto-load), R1 (semantic witness-filtered recall).

**Gap:** `resolveWorldMemoriesBook()` (plane1Context.js:84) resolves the book at write-time but never attaches it; `getEffectiveLorebookNames()` (utils.js:445) only returns the chat-bound book. So `<World> - Memories` isn't in ST's active WI set for inference.

**Step 1 — API spike (read-only).** Discover how an extension programmatically activates a lorebook for the current chat. Candidate calls to identify: global `selected_world_info` + a reload, the chat-lore `chat_metadata['world_info']` (METADATA_KEY), char-lore mapping, or an `eventSource` WI hook. Pick the mechanism that is **per-world + automatic + cross-chat** (leaning: toggle the world book into the active set on chat-enter, out on leaving the world).

> **PROVENANCE CORRECTION (2026-06-21, post-review).** An earlier draft of this step cited three extensions — `SillyTavern-WI-Bulk-Mover/index.js`, `SillyTavern-CharacterLibrary/modules/lorebook-manager.js`, `SillyTavern-Pathweaver/index.js` — as the source of the activation API. That citation was **wrong and not reproducible**: grepping all three on the host shows NONE reference `selected_world_info`, `onWorldInfoChange`, or `WORLDINFO_SETTINGS_UPDATED`. WI-Bulk-Mover only does `loadWorldInfo`/`reloadWorldInfoEditor`/`saveSettingsDebounced` (a WI *editor*, not an *activator*); `lorebook-manager.js` only touches the chat-bound link `chat_metadata.world_info`, not the global active set. The activation mechanism is nonetheless **correct**, but it was actually verified from two real sources, NOT those three:
> 1. **ST core `public/scripts/world-info.js`** (container-only, not host-readable — consistent with this phase's "ST internals aren't readable from the host" framing): `selected_world_info` (`:66`), `getGlobalLore()` (`:4414`), `onWorldInfoChange()` (`~:5653-5725`), `WORLDINFO_SETTINGS_UPDATED` (`events.js:41`).
> 2. **The sibling extension `SillyTavern-WorldScope/activation.js`** (host-readable, in the same container), which solves the identical "activate one world's books in the global layer, per-world, additive" problem and is the **proven pattern** Phase 4a now follows (see Step 2).

**Step 2 — Implement auto-attach.** Hooks on `CHAT_CHANGED` (`handleChatChanged`) **and before generation** (`GENERATION_STARTED`, non-dry-run) **and on startup** (`APP_READY`/`init`, for a chat already open at load): if `isTwoPlane()` and `resolveWorldMemoriesBook()` resolves, ensure `<World> - Memories` is in ST's active WI set for this chat; if the chat belongs to a *different* world, ensure the previous world's book is not lingering. Flag-gated (flag-off = byte-identical). Don't clobber a user's own chat-bound book (additive, not replace).

**Implementation notes (2026-06-21, post-review — these reflect the shipped 4a):**
- **Before-generation trigger IS wired.** All three arming sites route through one debounced helper `scheduleWorldBookActivation()` (index.js). This covers a world change that does NOT raise `CHAT_CHANGED` — e.g. the user re-tags the active character into a different ST folder mid-chat, or a group-membership change resolves a different `currentFolderName()`/roster world — which would otherwise leave the prior world's book active until the next chat switch. `GENERATION_STARTED` skips dry runs (token-count/preview passes). `applyWorldBookActivation()` is idempotent, so re-arming every generation is a no-op when the world is unchanged.
- **Activation mechanism = the proven WorldScope DOM-driven path.** PREFER `$('#world_info').val(indices).trigger('change')` (the COMPLETE next active set as `world_names` indices), letting ST's OWN `onWorldInfoChange()` rebuild `selected_world_info` from the multiselect, persist, and emit. This is **select2-safe**: a later genuine `change` (user opens the WI panel, select2 re-render, another extension) makes ST rebuild from the DOM's selected options — so going through `.val().change()` keeps the attached book from being dropped, the exact failure a direct in-place `push` + best-effort option-prop would hit. FALL BACK to in-place `splice`/`push` + `saveSettingsDebounced` + emit ONLY when the `#world_info` widget is not yet rendered (chat-open before the WI panel exists), so headless activation still works pre-panel.
- **Readiness + ordering guarded.** `applyWorldBookActivation()` bails (no resolve, no create) when `world_names` is empty — `CHAT_CHANGED` fires early/repeatedly and `world_names` is unreliable before `APP_READY` (per WorldScope's own notes). The 150 ms debounce collapses the early burst; a later `CHAT_CHANGED`/`GENERATION_STARTED` re-arms once `world_names` is hydrated. Together these prevent an early fire from creating a spurious `<World> - Memories` book.
- **Pure cores, host-tested.** `computeWorldBookActivation()` (additive delta, for the fallback + idempotency/logging) and `computeNextActiveWorlds()` (the COMPLETE next set, for the DOM-driven path) are pure and unit-tested (`tests/plane1Activation.test.js`); both preserve every non-memory book and only ever drop/add `* - Memories` names.

**Step 3 — LIVE VERIFY (user + me):** in the Shilpa+Aisha chat, a character references something only a memory knows ("fortress"/"sanctuary") → the model recalls it; confirm the witness gate (Aisha gets the shell, not the whisper content). Then open/create a *second* chat in 🏠 TWW2 → the same memory surfaces with no manual setup (X1/X3).

**Acceptance:** X1, X3, R1.

## 4b — Pre-switch tail flush (X2)

**Gap:** only `CHAT_CHANGED` exists (post-switch, in the *landing* chat). The design needs the **outgoing** chat's last ~20 unsummarized messages captured *before* leaving (SM's bug is exactly that CHAT_CHANGED runs in the chat you land on).

**Step 1 — find the hook.** Identify an ST pre-switch event available to extensions, or fall back to: on `CHAT_CHANGED`, read the *previous* chatId (tracked) and flush its tail; or hook chat save/unload. (Discover from ST events the other extensions use.)

**Step 2 — implement.** On switch, if the outgoing chat has unprocessed messages beyond its highwater, generate a memory for that tail into the world book (reuse the existing two-plane generation path), stamped with the outgoing chatId so it lands correctly. Flag-gated.

**Acceptance:** X2 (tail captured on the chat being left).

## 4c — Tier-bias verification + light tuning (R2/R3)

**Finding:** mostly **free**. Phase 3's disable-prune deactivates consolidated leaves, so the active retrievable set ≈ recent leaves + rollups — which *is* the tier-bias (recent=leaf-sharp, distant=rollup). ST's recency/order + vector similarity do the ranking.

**Step 1 — verify** the active set behaves as intended after a rollup (leaves disabled, arcs active).
**Step 2 — optional light tuning:** if recency isn't biasing well, set `order`/insertion so leaf tiers outrank rollups for recent context. Likely small.
**Step 3 — confirm R3:** witness-filter (characterFilter) is applied by ST regardless of budget pressure (verify ST never drops the filter to fit budget — it's a gate, not a ranked item, so it should hold).

**Acceptance:** R2, R3.
