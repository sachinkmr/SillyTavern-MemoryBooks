# SillyTavern 记忆书（STMB）如何工作

这是一份关于 STMB 如何工作的高层说明。它不是用来解释代码的。相反，这份文档说明的是：STMB 会组装哪些信息、按什么顺序发送，以及期望模型返回什么。

请把这份文档当作编写或修改 STMB 提示词时的参考。

## STMB 的 3 条主要提示流程

STMB 主要有三种工作流：

1. 记忆生成
2. 侧边提示
3. 整合

它们彼此相关，但期望的输出类型并不相同。

- 记忆生成期望严格的 JSON。
- 侧边提示通常期望干净的纯文本输出（可以使用 Markdown 或其他知识库条目格式，侧边提示中不要使用 JSON）。
- 整合也期望严格的 JSON，但它的结构和记忆不同。

## I. 记忆生成

当你创建一条记忆时，STMB 会发送一个组装好的提示，通常按这个顺序包含以下部分：

1. 选中的记忆提示或预设文本
   - 这是来自“摘要提示管理器”的指令块。
   - 它告诉模型要写什么类型的摘要，以及要返回什么 JSON 结构。
   - 像 `{{user}}` 和 `{{char}}` 这样的宏会在发送前先被解析。

2. 可选的先前记忆上下文
   - 如果这次运行被设置为包含先前记忆，它们会以只读上下文的形式插入。
   - 它们会被明确标记为上下文，而不是要再次总结的内容。

3. 当前场景转录
   - 选中的聊天范围会逐行格式化为 `说话者: 消息`。
   - 这才是模型需要真正转成记忆的场景。

大致结构如下：

```text
[记忆提示 / 预设指令]

=== PREVIOUS SCENE CONTEXT (DO NOT PROCESS) ===
[零条或多条更早的记忆]
=== END PREVIOUS SCENE CONTEXT - PROCESS ONLY THE SCENE BELOW ===

=== SCENE TRANSCRIPT ===
Alice: ...
Bob: ...
=== END SCENE ===
```

### 模型应该返回什么

我们期望返回一个 JSON 对象：

```json
{
  "title": "Short scene title",
  "content": "The actual memory text",
  "keywords": ["keyword1", "keyword2", "keyword3"]
}
```

最佳实践：

- 只返回这个 JSON 对象。
- 使用完全一致的键：`title`、`content` 和 `keywords`。
- 让 `keywords` 成为真正的 JSON 字符串数组。
- 标题要简短、易读。
- 关键词要具体、利于检索：地点、物品、专有名词、辨识度高的动作、标识符。

STMB 有时可以补救稍微有点乱的输出，但提示词不应该依赖这一点。

### 什么样的记忆提示是好的

好的记忆提示会清楚地做到四件事：

1. 告诉模型要写什么类型的记忆
   - 详细场景记录
   - 紧凑剧情概要
   - 极简回顾
   - 偏文学风格的记忆

2. 告诉模型什么信息重要
   - 剧情节点
   - 决策
   - 角色变化
   - 揭示
   - 结果
   - 与连续性有关的细节

3. 告诉模型什么该忽略
   - 通常是 OOC
   - 填充内容
   - 如果你想让记忆更紧凑，也可以忽略纯气氛闲聊

4. 明确告诉模型必须返回什么 JSON

### 什么样的记忆提示比较弱

弱提示通常会在以下某一方面失败：

- 只描述写作风格，却不说明 JSON 结构。
- 要求“有帮助的分析”或“想法”，而不是最终的记忆对象。
- 鼓励抽象关键词，而不是具体的检索词。
- 没有区分先前上下文和当前场景。
- 一次要求太多种输出格式。

### 为记忆编写提示词的实用建议

- 明确说明你要的是完整总结，还是更省 token 的总结。
- 如果你希望 `content` 里包含 Markdown，就直接说清楚。
- 如果你想让记忆更短，就限制正文，而不是改 JSON 结构。
- 如果你想让检索效果更强，就把提示空间花在关键词质量上，而不只是摘要风格。
- 把先前记忆当作连续性上下文，而不是要重写的素材。

## II. 侧边提示

侧边提示不是记忆。它们是追踪器 / 更新型提示，通常会写入或覆盖一个单独的知识库条目。这和记忆是完全不同的概念，这一点必须记住。

当一个侧边提示运行时，STMB 通常会按这个顺序组装以下部分：

1. 这个侧边提示本身的主指令文本
   - 这就是该追踪器的实际任务提示。
   - ST 标准宏，例如 `{{user}}` 和 `{{char}}`，会被解析。
   - 如果是手动运行，也可以插入自定义运行时宏。

2. 可选的旧条目
   - 如果这个侧边提示已经有保存内容，STMB 可以先把当前版本放进去。
   - 这样模型就可以更新现有追踪器，而不是每次都从头写。

3. 可选的先前记忆上下文
   - 如果模板要求包含先前记忆，STMB 会把它们作为只读上下文插入。

4. 编译后的场景文本
   - 这是当前场景中该追踪器应该响应的材料。

5. 可选的格式响应指导
   - 这不是强制解析结构。
   - 它只是你希望输出采用什么格式的附加说明。

大致结构如下：

```text
[侧边提示指令]

=== PRIOR ENTRY ===
[现有追踪器文本（如果有）]

=== PREVIOUS SCENE CONTEXT (DO NOT PROCESS) ===
[可选的先前记忆]
=== END PREVIOUS SCENE CONTEXT ===

=== SCENE TEXT ===
[编译后的场景文本]

=== RESPONSE FORMAT ===
[可选的格式指导]
```

### 模型应该返回什么

STMB 期望的是可以直接保存的纯文本。

这和记忆的关键区别在于：

- 侧边提示不需要 JSON。
- STMB 通常会把返回文本按原样保存。
- 如果你在侧边提示里要求 JSON，那么那段 JSON 也只是文本，除非你的工作流本来就依赖它。

这意味着，侧边提示的提示词应该追求可直接使用的最终输出，而不是方便解析的记忆 JSON。

### 什么样的侧边提示是好的

好的侧边提示应该是范围窄、结构稳定、易于更新的。

例如：

- 维护一个按重要性排序的“角色阵容”。
- 跟踪当前关系状态。
- 跟踪未解决的剧情线索，例如“情节要点”。
- 跟踪 `{{char}}` 目前对 `{{user}}` 的认知，例如“评估”。

最好的侧边提示措辞通常会做到这些：

1. 明确工作目标
   - “维护一个角色追踪表”
   - “更新当前关系状态表”
   - “维护一份未解决线索报告”

2. 说明是更新、替换还是追加
   - 这很重要，因为可能会包含旧条目文本。

3. 定义输出布局
   - 标题层级
   - 项目符号结构
   - 分区
   - 排序规则

4. 说明不要包含什么
   - 猜测
   - 重复项目
   - 过时信息
   - 对任务本身的说明性旁白

### 什么样的侧边提示比较弱

- 范围太宽，比如“追踪一切”。
- 从不说明旧条目应该被修订还是被重写。
- 要求链式思考或解释，而不是最终追踪器文本。
- 没有明确格式，导致追踪器会随着时间漂移。

### 为侧边提示编写提示词的实用建议

- 把侧边提示写成维护说明，而不是摘要提示。
- 默认模型可能会先看到当前追踪器，再看到新场景。
- 让每个追踪器只专注于一项工作。
- 使用“响应格式（可选）”字段来控制布局、分区名称和排序。

## III. 整合

整合会把较低层级的条目合并成较高层级的摘要。

例如：

- 把记忆整合成故事弧摘要
- 把故事弧摘要整合成章节摘要
- 把章节摘要整合成全书摘要

当整合运行时，STMB 通常会按这个顺序组装以下部分：

1. 选中的整合提示或预设文本
   - 它说明模型应该如何压缩这些源条目。
   - 它也定义了模型必须返回的 JSON 结构。

2. 可选的先前高层级摘要
   - 如果这个层级有要延续的先前摘要，它会先作为正典上下文插入。
   - 提示会告诉模型不要重写它。

3. 按时间顺序排列的已选低层级条目
   - 每个源条目都会带着标识符、标题和内容一起提供。
   - 这些才是模型要分组、压缩并转换成高层级摘要的材料。

大致结构如下：

```text
[整合提示 / 预设指令]

=== PREVIOUS ARC/CHAPTER/BOOK (CANON - DO NOT REWRITE) ===
[可选的先前高层级摘要]
=== END PREVIOUS ... ===

=== MEMORIES / ARCS / CHAPTERS ===
=== memory 001 ===
Title: ...
Contents: ...
=== end memory 001 ===

=== memory 002 ===
Title: ...
Contents: ...
=== end memory 002 ===
...
=== END ... ===
```

### 模型应该返回什么

STMB 期望的是一个如下结构的 JSON 对象：

```json
{
  "summaries": [
    {
      "title": "Short higher-tier title",
      "summary": "The consolidated recap text",
      "keywords": ["keyword1", "keyword2"],
      "member_ids": ["001", "002"]
    }
  ],
  "unassigned_items": [
    {
      "id": "003",
      "reason": "Why this item was left out"
    }
  ]
}
```

这里有个重要概念：

- 整合可以返回一个摘要，也可以返回多个摘要。
- `member_ids` 会告诉 STMB，每个返回摘要分别包含哪些源条目。
- `unassigned_items` 是模型表达“这个条目不适合归进我刚刚生成的摘要里”的方式。

### 什么样的整合提示是好的

好的整合提示会把三件事说明得很清楚：

1. 定义压缩目标
   - 一个故事弧
   - 一个或多个故事弧
   - 简洁但完整的回顾
   - 高压缩比的回顾

2. 定义选择逻辑
   - 保持时间顺序
   - 保持连续性
   - 合并相关条目
   - 把无关条目标为未分配

3. 非常明确地定义 JSON 结构

最好的整合提示还会告诉模型必须保留哪些内容：

- 主要剧情节点
- 转折点
- 已经立下的承诺
- 后果
- 未解决的线索
- 关系变化
- 对连续性至关重要的引语或标识符

### 什么样的整合提示比较弱

- 它要求回顾，却从不说明该如何给源条目分组。
- 它没有告诉模型该如何处理离群条目。
- 它没有要求 `member_ids`。
- 它要求自由散文，而不是整合 JSON 对象。
- 它过度关注文风，却没有把选择和分组规则说清楚。

### 为整合编写提示词的实用建议

- 告诉模型，你要的是一个连贯回顾，还是最少数量的若干个连贯回顾。
- 要求保持时间顺序。
- 要求明确处理剩余条目。
- 这里的关键词也要具体；高层级摘要同样需要可检索价值。

## 真正的提示词编写规则

为 STMB 写提示词时，不要只想：“我想让 AI 说什么？”

要想的是：

1. STMB 会在场景前面放入什么上下文？
2. 真正被分析的材料单位是什么？
3. 这条流程期望的是严格 JSON，还是最终纯文本？
4. 什么信息应该在之后的检索中保留下来？
5. 什么内容应该被忽略、压缩、保留，或延续到后面？

如果你的提示词把这五个问题答清楚了，它通常就能很好地配合 STMB 工作。

## FAQ 式说明

- “我能看到实际发给 AI 的内容吗？”
  可以。如果你想检查组装后的提示，请查看终端 / 日志输出。

- “如果我的提示词很弱，STMB 会强行把输出修好吗？”
  不太会。STMB 有时可以补救格式错误的 JSON，但它无法修复一个本来就要求错了东西的模糊提示。

- “重写提示词时，我最该先优化什么？”
  先优化返回格式。然后优化要保留哪些细节。风格排在这之后。
