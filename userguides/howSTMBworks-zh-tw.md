# SillyTavern 記憶書（STMB）怎麼運作

這是一份高層級說明，講的是 STMB 如何運作，不是拿來解釋程式碼的。重點是：STMB 會先組裝哪些資訊、照什麼順序送出，以及模型應該回傳什麼。

這份文件是拿來幫你替 STMB 寫提示詞、改提示詞的。

## STMB 的 3 種主要提示流程

STMB 有三種主要工作流程：

1. 記憶生成
2. 側邊提示詞
3. 整合

它們彼此相關，但不會期待相同的輸出。

- 記憶生成需要嚴格的 JSON。
- 側邊提示詞通常需要乾淨的純文字（可以用 Markdown 或其他知識庫條目格式，側邊提示詞裡不要用 JSON）。
- 整合也需要嚴格的 JSON，但格式和記憶不一樣。

## I. 記憶生成

當你建立記憶時，STMB 會送出一個組合好的提示詞，通常按這個順序包含以下部分：

1. 選到的記憶提示或預設文字
   - 這是來自摘要提示管理器的指令區塊。
   - 它會告訴模型要寫哪一種摘要，以及要回傳什麼 JSON 形狀。
   - 像 `{{user}}` 和 `{{char}}` 這類巨集會在送出前先展開。

2. 可選的先前記憶上下文
   - 如果這次執行有設定要帶入先前記憶，它們會先插入，作為只讀上下文。
   - 這些內容會清楚標示為上下文，不是要再摘要一次的主體。

3. 當前場景文字記錄
   - 選取的聊天範圍會逐行格式化成 `Speaker: message`。
   - 這才是模型應該轉成記憶的實際場景。

大致長這樣：

```text
[memory prompt / preset instructions]

=== PREVIOUS SCENE CONTEXT (DO NOT PROCESS) ===
[zero or more earlier memories]
=== END PREVIOUS SCENE CONTEXT - PROCESS ONLY THE SCENE BELOW ===

=== SCENE TRANSCRIPT ===
Alice: ...
Bob: ...
=== END SCENE ===
```

### 模型應該回傳什麼

我們期待的是一個 JSON 物件：

```json
{
  "title": "Short scene title",
  "content": "The actual memory text",
  "keywords": ["keyword1", "keyword2", "keyword3"]
}
```

最佳做法：

- 只回傳這個 JSON 物件。
- 使用精確的鍵名 `title`、`content`、`keywords`。
- `keywords` 要是字串陣列，不是字串。
- 標題要短，而且好讀。
- 關鍵詞要具體、方便檢索，例如地點、物件、專有名詞、獨特行動、識別碼。

STMB 有時候可以救回一些不太完整的輸出，但提示詞不要仰賴這件事。

### 什麼樣的記憶提示詞比較好

好的記憶提示詞會清楚做到這四件事：

1. 告訴模型要寫哪一種記憶
   - 詳細場景紀錄
   - 精簡摘要
   - 最小化回顧
   - 帶文學感的敘事記憶

2. 告訴模型哪些資訊重要
   - 劇情節點
   - 決策
   - 角色變化
   - 揭露
   - 結果
   - 對連續性有影響的細節

3. 告訴模型哪些內容可以忽略
   - 通常是 OOC
   - 填充內容
   - 純氣氛聊天，如果你想要更緊湊的記憶

4. 告訴模型要回傳什麼 JSON

### 什麼樣的記憶提示詞比較弱

弱的提示詞通常會卡在下面幾種問題裡：

- 它只描述寫作風格，卻沒有說清楚 JSON 形狀。
- 它要的是「分析」或「想法」，不是最後可用的記憶物件。
- 它鼓勵抽象關鍵詞，而不是具體的檢索詞。
- 它沒有區分先前上下文和當前場景。
- 它同時要求太多種輸出格式。

### 記憶的實用寫法

- 要明確說清楚摘要是要完整還是省 token。
- 如果你想在 `content` 裡用 Markdown，要直接講出來。
- 如果你要短記憶，限制正文長度，不要改 JSON 架構。
- 如果你重視檢索效果，就把提示詞空間花在關鍵詞品質上，不只是摘要風格。
- 先前記憶要當作連續性上下文，不要當成要重新改寫的素材。

## II. 側邊提示詞

側邊提示詞不是記憶。它們是追蹤器 / 更新提示詞，通常會寫入或覆寫另一個知識庫條目。這和記憶是完全不同的概念，一定要分清楚。

當側邊提示詞執行時，STMB 通常會按這個順序組裝內容：

1. 側邊提示詞的主要指令文字
   - 這是那個追蹤器真正要做的工作提示。
   - ST 標準巨集像 `{{user}}`、`{{char}}` 會展開。
   - 自訂執行階段巨集也可以在手動執行時插入。

2. 可選的先前條目
   - 如果那個側邊提示詞已經有保存過內容，STMB 可以把目前版本先放進去。
   - 這樣模型就能更新現有追蹤器，而不是每次都從頭寫。

3. 可選的先前記憶上下文
   - 如果範本有要求帶入先前記憶，STMB 會把它們插成只讀上下文。

4. 編譯後的場景文字
   - 這是追蹤器應該回應的目前場景材料。

5. 可選的回應格式指引
   - 這不是解析器強制要求的結構。
   - 它只是額外說明你希望的輸出格式。

大致長這樣：

```text
[side prompt instructions]

=== PRIOR ENTRY ===
[existing tracker text, if any]

=== PREVIOUS SCENE CONTEXT (DO NOT PROCESS) ===
[optional previous memories]
=== END PREVIOUS SCENE CONTEXT ===

=== SCENE TEXT ===
[compiled scene text]

=== RESPONSE FORMAT ===
[optional format guidance]
```

### 模型應該回傳什麼

STMB 期待的是可直接儲存的純文字。

這就是它和記憶的關鍵差別：

- 側邊提示詞不要 JSON。
- STMB 通常會原樣儲存回傳的文字。
- 如果你在側邊提示詞裡要求 JSON，那個 JSON 也只是文字，除非你自己的工作流程要再拿它做別的事。

所以側邊提示詞應該追求的是可直接使用的最終輸出，而不是適合解析器的記憶 JSON。

### 什麼樣的側邊提示詞比較好

好的側邊提示詞要窄、穩定，而且容易更新。

例如：

- 保留角色清單，並按重要性排序。
- 追蹤目前的關係狀態。
- 追蹤尚未解決的劇情線。
- 追蹤 `{{char}}` 目前對 `{{user}}` 的認知。

好的側邊提示詞通常會做到這幾件事：

1. 把工作定義清楚
   - 「維護角色名單追蹤器」
   - 「更新目前關係表」
   - 「保留未解決線索報告」

2. 說明是要更新、取代，還是附加
   - 這很重要，因為前一版條目可能會被先放進來。

3. 定義輸出排版
   - 標題
   - 項目符號結構
   - 區塊
   - 排序規則

4. 說明不要包含什麼
   - 推測
   - 重複項目
   - 過時資訊
   - 關於任務本身的敘述

### 什麼樣的側邊提示詞比較弱

- 範圍太大：「把所有東西都追蹤起來」。
- 沒說清楚舊條目是要修訂還是重寫。
- 要模型交代推理過程或解釋，而不是最後的追蹤器文字。
- 格式太模糊，導致追蹤器內容會越長越散。

### 側邊提示詞的實用寫法

- 把側邊提示詞寫成維護指令，不要寫成摘要提示詞。
- 預設模型可能會先看到目前追蹤器，再看到新場景。
- 每個追蹤器只做一件事。
- 用「回應格式」欄位來控制版面、區塊名稱和排序。

## III. 整合

整合是把較低層級的條目合成較高層級的摘要。

例如：

- 將記憶整合成較高層級摘要
- 將較低層級摘要整合成中間層摘要
- 將中間層摘要整合成最高層摘要

當整合執行時，STMB 通常會按這個順序組裝內容：

1. 選到的整合提示或預設文字
   - 它會說明模型要怎麼壓縮來源條目。
   - 也會定義模型應該回傳的 JSON 架構。

2. 可選的先前高層摘要
   - 如果這一層已經有前一版摘要要延續，它會先以正史內容放進來。
   - 提示詞會告訴模型不要重寫它。

3. 依時間順序排列的低層條目
   - 每個來源項目都會附上識別碼、標題和內容。
   - 這就是模型要分組、壓縮，並轉成更高層摘要的材料。

大致長這樣：

```text
[consolidation prompt / preset instructions]

=== PREVIOUS ARC/CHAPTER/BOOK (CANON - DO NOT REWRITE) ===
[optional previous higher-tier summary]
=== END PREVIOUS ... ===

=== MEMORIES / ARCS / CHAPTERS ===
=== memory 001 ===
Title: ...
Contents: ...
=== end memory 001 ===

=== memory 002 ===
Title: ...
Contents: ...
=== end memory 002 ===
...
=== END ... ===
```

### 模型應該回傳什麼

STMB 期待的是這種 JSON 物件：

```json
{
  "summaries": [
    {
      "title": "Short higher-tier title",
      "summary": "The consolidated recap text",
      "keywords": ["keyword1", "keyword2"],
      "member_ids": ["001", "002"]
    }
  ],
  "unassigned_items": [
    {
      "id": "003",
      "reason": "Why this item was left out"
    }
  ]
}
```

重點是：

- 整合可能回傳一個摘要，也可能回傳多個。
- `member_ids` 會告訴 STMB 哪些來源條目屬於哪一個回傳摘要。
- `unassigned_items` 是模型在說「這個條目不適合我剛剛做出的摘要」。

### 什麼樣的整合提示詞比較好

好的整合提示詞會把這三件事做好：

1. 定義壓縮目標
   - 一個 arc
   - 一個或多個 arc
   - 盡量簡潔但完整的回顧
   - 強力壓縮的回顧

2. 定義選取邏輯
   - 保持時間順序
   - 保住連續性
   - 合併相關項目
   - 把不相關的項目留在未分配區

3. 清楚定義 JSON 結構

最好的整合提示詞也會告訴模型要保留什麼：

- 主要節點
- 轉折點
- 承諾
- 結果
- 未解決線索
- 關係變化
- 對連續性有影響的關鍵引文或識別碼

### 什麼樣的整合提示詞比較弱

- 它只說要回顧，卻沒說怎麼把來源條目分組。
- 它沒告訴模型遇到離群項目時要怎麼辦。
- 它沒要求 `member_ids`。
- 它要的是自由散文，不是整合用的 JSON 物件。
- 它太重視風格，卻沒有把選取和分組說清楚。

### 整合的實用寫法

- 要告訴模型你想要的是一份連貫摘要，還是最少數量、但仍然連貫的摘要。
- 要求時間順序。
- 要求明確處理剩餘條目。
- 關鍵詞也要保持具體；更高層摘要仍然需要可檢索性。

## 真正的提示詞規則

替 STMB 寫提示詞時，不要只想：「我希望 AI 說什麼？」

你要想的是：

1. STMB 會把哪些上下文放在場景前面？
2. 被分析的實際單位是什麼？
3. 這條路徑要的是嚴格 JSON 還是最終純文字？
4. 哪些資訊之後還要能被檢索？
5. 模型應該忽略、壓縮、保留，還是延續什麼？

如果你的提示詞能把這五件事講清楚，通常就能和 STMB 配合得很好。

## FAQ 式補充

- 「我可以看到實際送給 AI 的內容嗎？」
  可以。如果你想檢查組裝後的提示詞，請看終端機 / 記錄輸出。

- 「如果我的提示詞很弱，STMB 會自動幫我修成好輸出嗎？」
  不會完全如此。STMB 有時候可以修補格式不完整的 JSON，但它無法補救一個本來就問錯方向的模糊提示詞。

- 「重寫提示詞時，應該先優化什麼？」
  先優化回傳格式。再優化要保留哪些細節。風格放在後面。
