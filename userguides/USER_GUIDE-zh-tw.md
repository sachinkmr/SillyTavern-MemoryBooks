**將漫長的聊天對話整理成有條理、可搜尋的記憶！**

需要讓機器人記住事情，但聊天內容太長，已經超出上下文能負荷的範圍？想要自動追蹤重要劇情點，又不想自己手動做筆記？ST Memory Books 就是為此而設計的。它會觀察您的聊天，建立智慧摘要，讓您不再輕易忘記故事中發生過的事。

## 🚀 快速開始（5 分鐘建立您的第一筆記憶！）

**第一次使用 ST Memory Books？** 只要幾個步驟，就可以先把自動記憶跑起來。

### 第一步：找到擴充功能

- 在聊天輸入框旁邊找到魔杖圖示（🪄）
- 點開它，然後點選 **Memory Books**
- 您會看到 ST Memory Books 的控制面板

### 第二步：開啟自動模式

- 在控制面板中找到 **自動建立記憶摘要**
- 把它打開
- 把 **自動摘要間隔** 設成 **20-30 則訊息**（這是很好的起始值）
- 一開始先把 **自動摘要緩衝區** 維持在低值（`0-2` 很適合新手）
- 先手動建立一筆記憶，讓這個聊天完成初始化
- 這樣就可以了！🎉

### 第三步：像平常一樣聊天

- 繼續正常聊天
- 當新增訊息累積到大約 20-30 則後，ST Memory Books 會自動：
  - 使用自上次處理完成之後的新訊息
  - 請您的 AI 寫出摘要
  - 把摘要存進記憶集合
  - 在完成時顯示通知

**恭喜！** 您現在已經有自動記憶管理了。之後就比較不容易忘記幾個章節之前發生過什麼。

## 💡 ST Memory Books 實際上做了什麼

您可以把 ST Memory Books 想成聊天對話的 **AI 私人圖書管理員**。

### 🤖 **自動摘要**
*「我不想一直管它，只想讓它自己運作」*

- 在背景監控聊天
- 每隔一段訊息數自動建立記憶
- 適合長篇角色扮演、創作寫作、持續進行的故事

### ✋ **手動建立記憶**
*「我想自己決定哪些內容要被保存」*

- 用簡單的箭頭按鈕（► ◄）標記重要場景
- 在重要時刻手動建立記憶
- 很適合保存關鍵劇情點或角色成長片段

### 📊 **側邊提示詞與智慧追蹤器**
*「我想追蹤關係、劇情線、狀態或數值」*

- 可重複使用的提示片段，幫助記憶生成
- 內建範本庫，可直接拿來用
- 可以自訂 AI 提示，追蹤任何您想追蹤的內容
- 會自動更新記分板、關係狀態、劇情摘要等資訊
- 例如：「誰喜歡誰？」「目前任務狀態」「角色心情追蹤」

### 📚 **記憶集合**
*所有記憶存放的地方*

- 會自動整理，方便搜尋
- 與 SillyTavern 內建 lorebook 系統整合
- 您的 AI 可以在後續聊天中參考過往記憶

<details>
<summary><strong>🔄 「設定後就交給它」(推薦新手)</strong></summary>

**適合您，如果您想要：** 幾乎不用手動干預，讓系統自己穩定運作

**運作方式：**
1. 開啟 `自動建立記憶摘要`
2. 把 `自動摘要間隔` 設成適合您聊天節奏的數值
3. 如果想讓生成稍微延後，可以加一點 `自動摘要緩衝區`
4. 先手動建立一筆記憶完成初始化，之後就讓它自動跑

**您會得到：**
- 幾乎不需要手動處理
- 穩定持續的記憶建立
- 不容易漏掉重要劇情
- 單人聊天與群組聊天都能用

**小建議：** 可以先從 30 則訊息開始，再依照聊天速度調整。聊天節奏很快時可以往上調，聊天節奏很慢、內容很細時可以往下調。

</details>

<details>
<summary><strong>✋ 「手動控制」(只保存您想保存的內容)</strong></summary>

3. 在場景最後一則訊息按下 ◄

**您會得到：**
- 對記憶內容有完整控制權

**小建議：** 箭頭按鈕通常會在聊天載入後幾秒內出現。如果一開始看不到，稍等一下或重新整理頁面。

- 更快速的記憶建立流程
- 比較容易批次處理
- 更方便接進自己的工作流

## ✂️ 剪輯到記憶書

當您只想保存一句重要台詞或一個事實，而不需要建立完整場景記憶時，使用 **剪輯到記憶書**。在聊天中選取文字，點擊浮動剪刀按鈕，然後選擇既有剪輯條目或建立新條目。

不確定該用剪輯還是 Side Prompt？請看 [剪輯 vs 側邊提示詞](#clips-vs-side-prompts)。

### 什麼時候該用剪輯？

剪輯最適合保存您希望 AI 記住的小事實，例如：

- 角色偏好
- 承諾或秘密
- 關係細節
- 寵物、地點、物品或反覆出現的細節
- 不需要完整記憶摘要的快速備忘

如果是較大的場景，請改用正常的記憶建立。

### 剪輯如何運作

1. 選取您想保存的句子或短語。
2. 點擊浮動剪刀按鈕。
3. 選擇一個既有剪輯條目，或建立新條目。
4. 檢查條目預覽。
5. 保存剪輯。

剪輯條目是一般 lorebook 條目，但會標記為 `[STMB Clip]`。例如：

```txt
Seraphina Healed Me [STMB Clip]
```

在條目內部，STMB 會用清楚的區段格式保存內容：

```md
=== Seraphina Healed Me ===

- Seraphina healed my wounds with magic.

=== END Seraphina Healed Me ===
```

### 建立或重新命名剪輯條目

建立新剪輯條目時，條目標題也會成為區段標題。您可以在剪輯時重新命名條目，STMB 會同步更新區段標題。

新的剪輯條目可以是：

- **常駐啟用**：適合應該一直可用的事實
- **關鍵字觸發**：適合只有在相關詞出現時才需要的事實

當剪輯只和某個主題、角色、地點、寵物、物品或關係有關時，使用關鍵字會更合適。

### 浮動剪刀按鈕

浮動剪刀按鈕只會在您選取聊天內文字後出現。您可以在 Memory Books 主彈窗中開啟或關閉這個按鈕。

### 檢查較長的剪輯條目

如果剪輯條目變長，STMB 可能會提醒您檢查它。您可以自己編輯，也可以使用 **壓縮**，讓 AI 把剪輯、Side Prompt 或 STMB 記憶條目改寫得更節省 token，然後再決定是否替換原文。

## ✂️ 剪輯 vs 側邊提示詞

剪輯和 Side Prompts 都會把資訊保存到您的記憶書裡，但它們不是同一種用途。

簡單規則：**剪輯保存一個具體事實。Side Prompt 維護一個會變化的追蹤器。**

| **剪輯** | **Side Prompts** |
|---|---|
| 把選取的聊天文字保存到記憶書條目中。 | 請 AI 閱讀聊天並更新追蹤器條目。 |
| 最適合一個明確事實、台詞、承諾、偏好、物品或筆記。 | 最適合會隨時間變化的資訊，例如關係狀態、任務進度、庫存或未解決劇情線。 |
| 您選擇確切文字，STMB 保存您選取的內容。 | AI 會解讀聊天，然後寫入或更新追蹤器。 |
| 當事實已經很明確、不需要分析時使用。 | 當 AI 需要比較、總結或根據多則訊息更新狀態時使用。 |
| 通常只會在您手動新增另一條剪輯時增長。 | 可以隨著故事變化反覆更新。 |
| 可以理解為：「把這條筆記釘住。」 | 可以理解為：「讓這個區塊保持更新。」 |

適合做剪輯的例子：

- `Aiko 喜歡蜂蜜茶。`
- `Theron 承諾不再對她說謊。`
- `Colt 叫她 Boss。`

適合做 Side Prompt 的例子：

- 關係狀態
- 目前任務進度
- 庫存與資源
- NPC 名錄
- 未解決劇情線

如果您只需要記住一個細節，用剪輯。如果您需要持續更新的追蹤器，用 Side Prompt。

## 🔎 主題剪輯

主題剪輯用於從你已經建立的記憶中，製作一條聚焦的「關於這個主題」記憶條目。

可以把它理解為讓 STMB：

> 「閱讀我保存的記憶，然後為這個人物、地點、關係、劇情線、物品、秘密或主題建立一條有用的條目。」

它仍然是剪輯式條目，但你不是在剪輯選取的聊天文字。STMB 會使用既有記憶條目作為來源。

簡單規則：**剪輯保存選取的文字。主題剪輯從已保存記憶中整理相關細節。Side Prompt 持續維護追蹤器。**

### 什麼時候使用主題剪輯

當你的記憶書中已經有多條記憶，並且你想要一條更容易觸發的特定主題條目時，使用主題剪輯。

適合的例子：

- 反覆出現的 NPC
- 兩個角色之間的關係
- 一個謎團或調查
- 一個地點
- 一個派系
- 角色的能力、傷勢、承諾、秘密或偏好
- 跨多個場景出現的劇情線

主題範例：

```txt
Seraphina
{{user}} 的魔法
Alex 和 Mira 的關係
Black Harbor 調查
銀鑰匙
```

### 什麼時候不要使用主題剪輯

不要在這些情況下使用主題剪輯：

- 你只想保存聊天中選取的一句話 — 使用**剪輯到記憶書**
- 你想要一個在未來記憶執行中自動更新的追蹤器 — 使用 **Side Prompt**
- 你想縮短一條過長條目 — 使用**壓縮**
- 你想把多條記憶合併成更高層級的總結 — 使用**總結整合**

### 如何使用主題剪輯

1. 打開 Memory Books 彈窗。
2. 點擊 **🔎 主題剪輯**。
3. 選擇**來源記憶書**。
4. 輸入**主題**。
   - 這是 AI 應該聚焦的對象。
   - 盡量具體。
5. 輸入**關鍵字**。
   - 這些會成為世界書條目的啟用關鍵字。
   - 如果留空，STMB 會使用主題。
6. 選擇模式：
   - **建立新的主題剪輯**會建立新的 `[STMB Clip]` 條目。
   - **更新現有條目**會更新既有剪輯條目。
7. 選擇**生成設定檔**。
   - 這決定由哪個 AI 連線/模型來撰寫草稿。
8. 可選：如果想更改發送給 AI 的說明，點擊**編輯主題剪輯提示**。
9. 點擊**生成草稿**。
10. 檢查生成的草稿。
11. 如有需要，編輯草稿。
12. 點擊**儲存主題剪輯**。

STMB 不會自動保存草稿。只有點擊**儲存主題剪輯**後，世界書才會更改。

### 建立新的主題剪輯

建立新的主題剪輯時，STMB 會建立一條剪輯式世界書條目。

例如，如果你的主題是：

```txt
Seraphina
```

條目標題會類似：

```txt
關於 Seraphina [STMB Clip]
```

條目中的可見部分會使用與普通剪輯條目相同的剪輯包裝格式。

### 更新既有主題剪輯

主題剪輯也可以更新既有的 `[STMB Clip]` 條目。

如果你已經有這樣的條目：

```txt
關於 Seraphina [STMB Clip]
```

而上次更新之後又新增了記憶，這會很有用。

主題剪輯更新成功保存後，STMB 會在該條目上保存一小段執行歷史。它包含本次執行使用過的來源記憶。下一次更新時，STMB 可以利用這段歷史，只查找新的或已變更的來源記憶，而不是重新讀取所有內容。

這樣可以讓更新更小，也能避免反覆把相同的舊記憶發送給 AI。

### 從所有來源記憶重建

更新既有主題剪輯時，你可能會看到**從所有來源記憶重建**。

普通更新時請保持關閉。STMB 會盡可能只使用新的或已變更的來源記憶。

在這些情況下開啟：

- 既有主題剪輯已經嚴重過時
- 你更改了主題剪輯提示
- 你大幅更改了主題或關鍵字
- 你想讓 AI 重新考慮該主題的所有已保存記憶
- 該條目還沒有有用的執行歷史

### 它使用哪些來源條目？

主題剪輯使用所選記憶書中已確認的 STMB 記憶條目。

它不會使用：

- 普通剪輯條目
- Side Prompt 追蹤器條目
- 不由 STMB 管理的普通世界書條目

這樣可以讓主題剪輯專注於 STMB 能安全識別的記憶。

### 使用主題剪輯的好習慣

使用聚焦的主題。

較好：

```txt
Alex 和 Mira 的關係
```

不太有用：

```txt
關於整個故事的一切
```

較好：

```txt
銀鑰匙
```

不太有用：

```txt
重要物品
```

主題越具體，AI 越容易判斷哪些內容屬於這個主題，哪些不屬於。

### 編輯提示

主題剪輯提示可以編輯。

預設提示會要求 AI：

- 只擷取與主題相關的資訊
- 避免無關事件
- 保留姓名、關係、偏好、承諾、秘密、限制和未解決問題
- 發現衝突時說明衝突，而不是悄悄選擇一個版本
- 更新既有剪輯內容，但不要重複
- 不要編造缺失細節

提示必須包含：

```txt
{{SOURCE_MEMORIES}}
```

沒有這個 placeholder，STMB 就不知道應該把來源記憶放在哪裡。

其他支援的 placeholder 包括：

```txt
{{MODE}}
{{TOPIC}}
{{KEYWORDS}}
{{EXISTING_CLIP}}
{{EXISTING_ENTRY_CONTENT}}
{{SOURCE_MEMORIES}}
```

如果你的自訂提示不再好用，請使用 **Reset to Default**。

## 🙈 節省 Token：隱藏 / 取消隱藏訊息

在長聊天中，最簡單的節省 token 方法之一，就是把**已經做成記憶的舊訊息隱藏起來**。

### 「隱藏」是什麼意思？

隱藏訊息 **不是刪除**。原始聊天仍然存在，記憶也仍然在 lorebook 裡。只是那些原始訊息不再直接送給 AI。

### 什麼時候會有幫助？

隱藏 / 取消隱藏功能很適合以下情況：

- 聊天已經變得很長
- 那些訊息已經做成記憶
- 您想讓聊天畫面更乾淨

### 建立記憶後自動隱藏

STMB 可以在建立記憶後自動隱藏訊息。您可以選擇：

- **不自動隱藏**：全部保持可見（您也可以自己用 `/hide x-y` 手動隱藏）
- **自動隱藏直到最後一筆記憶為止的所有訊息**：隱藏所有已被記憶覆蓋的內容
- **僅自動隱藏最後一筆記憶中的訊息**：只隱藏最近剛處理完的範圍

您也可以用 **保留可見訊息數** 來決定有多少最近訊息要保持顯示。

### 建立記憶前先取消隱藏

**建立記憶前取消隱藏訊息** 這個設定會在產生記憶前，暫時對選取範圍執行 `/unhide X-Y`。  
如果您常常重做記憶，或聊天中有不少已隱藏內容，這個設定會很有用。

### 新手推薦設定

Aiko 自己偏好的做法是：

- 使用 **自動隱藏直到最後一筆記憶為止的所有訊息**
- 保留 **2 則訊息** 可見
- 開啟 **建立記憶前取消隱藏訊息**

## 🧭 壓縮 vs 總結整合

名字有點像，但它們做的是不同工作。

簡單規則：**壓縮清理一個條目。總結整合把多個記憶合併成更高層級的回顧。**

| **壓縮** | **總結整合** |
|---|---|
| 讓一個既有的 STMB 管理條目變短。 | 把多個記憶或總結合併成一個更高層級的回顧。 |
| 一次處理一個剪輯、Side Prompt 條目或 STMB 記憶條目。 | 從多個已選擇的記憶 / 總結條目生成。 |
| 適合條目仍然有用，但太長、重複或佔用太多上下文的時候。 | 適合舊場景記憶堆積起來，應該變成 Arc、Chapter、Book、Legend、Series 或 Epic 總結的時候。 |
| 以更節省 token 的形式重寫所選條目。 | 從所選來源條目建立一個新的總結條目。 |
| 應該保留既有事實，並移除臃腫內容。 | 應該保留更大的連續性脈絡，並減少逐場景細節。 |
| 不會從原始聊天建立新記憶。 | 不會單獨壓縮某個臃腫條目。 |
| 可以理解為：「修剪這一個條目。」 | 可以理解為：「把這些記憶卷成一條回顧。」 |

兩個工具都需要先審閱。STMB 會先顯示 AI 寫出的內容，再保存或替換任何東西。

## 🌈 總結整合

總結整合可以把較舊的 STMB 記憶壓縮成更高層級的回顧摘要，讓長篇故事更容易管理。

### 問：什麼是總結整合？

**答：** STMB 不只是一直做場景等級的記憶。它也可以把既有記憶或既有總結，再往上整合成更精簡的回顧。第一層是 **Arc**，更長篇故事還可以往上做成：

- Arc
- Chapter
- Book
- Legend
- Series
- Epic

### 問：為什麼要用？

**答：** 總結整合在以下情況特別有用：

- 您的記憶列表已經很長
- 舊條目已經不需要逐場景的細節
- 您想降低 token 使用量，但又不想失去故事連貫性
- 您想要更高層級、更乾淨的劇情回顧

### 問：它會自動執行嗎？

**答：** 不會。總結整合仍然需要您的確認。

- 您可以隨時從主視窗手動打開 **整合記憶**
- 您也可以啟用 **當某個層級準備好時提示進行整合**
- 當某個已選擇的目標層級，達到其已儲存的最小來源數量時，STMB 會顯示 **yes/later** 確認提示
- 按下 **Yes** 只會幫您打開整合視窗，並預先選好該層級；它不會靜默自動跑完

### 問：我要怎麼使用？

**答：** 建立一筆整合摘要時：

1. 在 STMB 主視窗中點擊 **整合記憶**
2. 選擇目標摘要層級
3. 挑選您要納入的來源條目
4. 如有需要，可在新摘要建立後停用來源條目
5. 點擊 **Run**

如果 AI 回傳了不理想的整合結果，STMB 也可以先讓您檢查 / 修正，再重新提交。

### 什麼會被整合？什麼不會？

總結整合處理的是 **STMB 記憶** 與 **STMB 摘要**。

也就是說：

- 一般記憶可以被整合成更高層級的摘要
- 更高層級的摘要之後也可以再往上整合成更大的回顧

Side Prompt 不一樣。

**Side Prompt 是追蹤器條目**，不是記憶摘要條目。它們的用途是持續維護某些狀態資訊，例如：

- 關係狀態
- 目前目標
- 記分板
- 世界狀態
- 劇情線追蹤

用最直白的方式說：

- **記憶** =「這個場景發生了什麼？」
- **Side Prompt** =「這個東西現在是什麼狀態？」
- **總結整合** =「把很多記憶濃縮成更大的回顧摘要」

### 為什麼這點很重要？

因為有些使用者會以為追蹤器條目也會一起被捲進 Arc 或 Chapter 總結裡。

它們 **不會**。

如果您想讓某個內容參與整合，它必須是一般的 STMB 記憶 / 摘要條目，而不只是 Side Prompt。

### 簡單判斷原則

請這樣分：

- **記憶**：用來保存場景回顧
- **Side Prompt**：用來維護持續更新的追蹤資訊
- **總結整合**：用來把舊記憶壓縮成更高層級的回顧

## 🎨 追蹤器、側邊提示詞與範本（進階功能）

**側邊提示詞** 是背景追蹤器，用來維持故事中持續變動的資訊。  
它們會在記憶建立流程旁邊一起運作，並逐步更新獨立的側邊提示詞故事書條目。您可以把它想成是**一直看著故事、負責幫您記住某些狀態的助手**。

如果您只是想保存一段已選取的固定事實，請使用 [剪輯到記憶書](#clip-to-memory-book)。Side Prompts 更適合重複或持續追蹤的資訊。

### 🚀 **用範本快速開始**

1. 打開 Memory Books 設定
2. 點擊 **側邊提示詞**
3. 瀏覽 **範本庫**，挑選適合您故事的內容：

   - **Character Development Tracker** - 追蹤角色個性與成長變化
   - **Relationship Dynamics** - 追蹤角色之間的關係
   - **Plot Thread Tracker** - 追蹤持續進行中的劇情線
   - **Mood & Atmosphere** - 追蹤情緒與氛圍
   - **World Building Notes** - 追蹤世界觀與設定細節
4. 啟用您要使用的範本（之後隨時可以再修改）
5. 如果範本使用自動觸發，STMB 就會在記憶建立過程中一起更新該 side-prompt 條目

[Scribe：逐步示範如何啟用自動側邊提示詞](https://scribehow.com/viewer/How_to_Enable_Side_Prompts_in_Memory_Books__fif494uSSjCmxE2ZCmRGxQ)

### ⚙️ **側邊提示詞怎麼運作**

- **背景追蹤器**：它們會安靜地在背景更新資訊
- **不干擾主體**：不會直接改動您的主要 AI 設定或角色提示
- **可依聊天分開控制**：不同聊天可以啟用不同追蹤器
- **以範本為基礎**：可以使用內建範本，也可以自己建立
- **可自動也可手動**：標準範本可以自動執行；帶有自訂 runtime 巨集的範本則只能手動執行
- **巨集支援**：`Prompt`、`Response Format`、`Title` 與關鍵字欄位都可以使用標準 ST 巨集，例如 `{{user}}`、`{{char}}`
- **Runtime 巨集**：非標準 `{{...}}` 會變成執行時必填輸入，例如 `{{npc name}}="Jane Doe"`
- **可回傳純文字**：側邊提示詞不一定要回傳 JSON
- **覆寫式更新**：側邊提示詞會持續更新自己的追蹤條目，而不是每次都新增一筆新的連續記憶

### 🛠️ **管理側邊提示詞**

- **側邊提示詞管理器**：建立、編輯、複製、整理追蹤器
- **啟用 / 停用**：可隨時打開或關閉追蹤器
- **匯入 / 匯出**：方便分享或備份範本
- **狀態檢視**：查看目前聊天中哪些追蹤器已啟用、何時運作
- **安全檢查**：如果範本包含自訂 runtime 巨集，STMB 會在儲存 / 匯入時移除其自動觸發設定，並顯示警告 toast

### 💡 **範本示例**

- Side Prompt 範本庫（可直接匯入這份 JSON）：
  [SidePromptTemplateLibrary.json](../resources/SidePromptTemplateLibrary.json)

一些常見提示思路：

- 「追蹤重要對話與角色互動」
- 「持續更新目前任務狀態」
- 「記下新出現的世界觀細節」
- 「追蹤角色 A 與角色 B 的關係」

### 🔧 **建立自訂側邊提示詞**

1. 打開側邊提示詞管理器
2. 點擊 **新建**
3. 寫下簡短且清楚的指令  
   （例如：「每個場景都記下天氣狀態」）
4. 如有需要，可加入標準 ST 巨集，例如 `{{user}}` 或 `{{char}}`
5. 如果您加入自訂 runtime 巨集，例如 `{{location name}}`，就必須用 `/sideprompt "Name" {{location name}}="value"` 手動執行
6. 儲存並啟用
7. 如果它使用的是自動觸發，就會在之後持續更新；否則就要在需要時手動執行

### 💬 **小建議**

側邊提示詞 最適合拿來做**小而專注**的事情。  
與其寫成「追蹤所有東西」，通常更好的做法是像「追蹤主角之間的曖昧張力」這樣具體。

### ⌨️ **手動 `/sideprompt` 語法**

使用：
`/sideprompt "Name" {{macro}}="value" [X-Y]`

範例：

- `/sideprompt "Status" 10-20`
- `/sideprompt "NPC Directory" {{npc name}}="Jane Doe" 40-50`
- `/sideprompt "Location Notes" {{place name}}="Black Harbor" 100-120`

注意：

- 側邊提示詞名稱必須加引號
- Runtime 巨集的值也必須加引號
- 選好側邊提示詞之後，slash 指令自動完成會提示您補上需要的 runtime 巨集
- 如果範本包含自訂 runtime 巨集，STMB 會強制讓它只能手動執行，並移除自動觸發
- `X-Y` 是可選的；如果省略，STMB 會使用該 Side Prompt 上次更新之後的訊息

### 🧠 使用 Regex 擴充功能做進階文字控制

**想更精細地控制 STMB 傳給 AI 的文字，以及 AI 回傳後 STMB 如何處理它？**  
STMB 可以在生成前與儲存前，執行您挑選的 Regex 腳本。

這對以下情況很有用：

- 清除 AI 回應裡重複、沒用的雜訊
- 在生成前統一名稱或術語
- 在 STMB 預覽 / 解析前先整理文字格式

#### **現在的用法**

1. 先在 SillyTavern 的 **Regex** 擴充功能裡建立您要的腳本
2. 在 STMB 中打開 **使用正則表達式（進階）**
3. 點擊 **📐 設定正則表達式…**
4. 選擇 STMB 要執行哪些腳本：
   - 在送給 AI 之前執行
   - 在加入 lorebook 之前執行

#### **重要行為**

- STMB 的 Regex 選擇是由 **STMB 自己管理**
- 某個腳本即使在 Regex 擴充功能中被停用，只要它在 STMB 內被選中，仍然可能執行
- STMB 的送出前 / 儲存前兩側都支援多選

#### **快速示例**

如果您的模型總是加上一句 `(OOC: I hope this summary is helpful!)`，您可以：

1. 建立一個會移除這段文字的 Regex 腳本
2. 在 STMB 中打開 **使用正則表達式（進階）**
3. 點開 **📐 設定正則表達式…**
4. 把這個腳本加進 **incoming** 選單

之後 STMB 就會在預覽或儲存前，先把這段文字清掉。

## 🧹 壓縮

壓縮適用於某個由 STMB 管理的 lorebook 條目仍然有用、但已經變得太長或重複的情況。您不用手動刪改，可以讓 AI 把這個條目重寫成更節省 token 的版本。

不確定該用壓縮還是總結整合？參考上面的簡單規則：**壓縮清理一個條目。總結整合把多個記憶合併成更高層級的回顧。**

這是一個 **先審閱** 的工具。STMB 會在替換任何內容之前顯示原始內容與壓縮草稿。

### 什麼可以被壓縮？

壓縮可以列出所選記憶書中的這些條目：

- 剪輯條目
- 側邊提示詞追蹤條目
- STMB 記憶條目

它不會顯示不由 STMB 管理的一般 lorebook 條目。

### 如何使用壓縮

1. 打開 Memory Books 彈窗。
2. 點擊 **📝 壓縮**。
3. 選擇您要檢查的 **記憶書**。如果目前聊天已經有記憶書，它可能會自動選取。
4. 選擇一個 **壓縮設定檔**。這會決定由哪個 AI 連線 / 模型重寫條目。
5. 選用：如果想修改重寫指令，請點擊 **編輯壓縮提示**。
6. 在表格中找到條目，然後點擊 **壓縮條目**。
7. 檢查結果：
   - **原始內容** 顯示目前已儲存的內容。
   - **壓縮草稿** 顯示 AI 重寫結果。
   - 兩者都會顯示估算 token 數。
8. 視需要編輯壓縮草稿。
9. 選擇一個操作：
   - **替換為壓縮版本**：把草稿儲存並覆蓋原條目。
   - **複製壓縮草稿**：只複製草稿，不儲存。
   - **取消**：保持條目不變。

STMB 不應該靜默替換原文。如果您沒有點擊 **替換為壓縮版本**，lorebook 條目會保持原樣。

### 編輯壓縮提示

壓縮提示控制 AI 如何重寫條目。內建提示刻意偏保守：保留重要事實、姓名、代名詞、巨集、包裝標題和結尾標記；移除重複和低價值措辭；不要編造內容。

此提示支援這些佔位符：

- `{{ENTRY_CONTENT}}` — 目前條目內容。此項必需。
- `{{ENTRY_KIND}}` — 條目類型，例如 Clip、SidePrompt 或 Memory。
- `{{ENTRY_TITLE}}` — 條目標題。

如果自訂提示開始表現不好，請使用 **重設為預設值**。

### 適合的用途

適合用壓縮處理：

- 較長的剪輯條目
- 逐漸重複自己的側邊提示詞追蹤器
- 內容正確但臃腫的記憶條目
- 一直啟用、正在消耗過多 token 的條目

不要用它來：

- 從聊天中建立新記憶
- 添加新事實
- 修補條目裡原本不存在的連續性
- 編輯 STMB 之外的一般 lorebook 條目

壓縮是清理工具，不是記憶生成工具。

- 確認 STMB 已啟用，而且 **Memory Books** 出現在擴充功能選單中
- 如果自動摘要沒有觸發，先確認您已手動建立過一筆記憶，並檢查 interval / buffer 是否合理
- 如果記憶無法儲存，確認目前聊天有綁定 lorebook，或已啟用 **若無 lorebook 則自動建立**
- 如果 Regex 行為看起來不對，請檢查 **📐 設定正則表達式…** 裡的選項，而不只是看 Regex 擴充功能本身
- 如果整合提示沒有出現，請確認 **當某個層級準備好時提示進行整合** 已打開，而且目標層級有包含在 **自動整合層級** 裡

## 🚫 ST Memory Books 不做什麼

- **它不是通用 lorebook 編輯器**：這份指南主要聚焦在 STMB 建立的條目。一般 lorebook 編輯請使用 SillyTavern 內建的 lorebook 編輯器。
